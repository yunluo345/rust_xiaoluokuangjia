# 提取标签逻辑图

## 配置文件：ai.json → Aipeizhi.ribaoshengcheng

提取标签的所有行为都由配置文件 `peizhi/ai.json` 中的 `ribaoshengcheng` 字段驱动：

```
ribaoshengcheng:
├── xinxi_yingshe（信息类别 → 模板占位符映射）
│   ├── "时间"           → "riqi"
│   ├── "地名"           → "didian"
│   ├── "工作内容"       → "gongzuoneirong"
│   ├── "对话内容"       → "duihuaneirong"
│   ├── "后续计划"       → "houxujihua"
│   ├── "备注"           → "beizhu"
│   ├── "对方派遣人员"   → "duifangpaiqianrenyuan"
│   ├── "客户公司"       → "kehugongsi"
│   └── "自己名称"       → "zijimingcheng"
│
└── shuchu_moban（日报输出格式模板，使用上面的占位符）
```

配置在提取标签流程中的作用：

- `xinxi_yingshe` 的 **keys**（时间、地名、工作内容...）= 允许的标签类别 = 必需的标签类别
- 构建系统提示词时，把所有 keys 拼接告诉 AI "只能提取这些类别"
- 验证阶段用 keys 过滤非法类别、检查必需类别是否齐全
- 后续日报生成阶段用 **values**（riqi、didian...）替换模板中的占位符

## 入口一：tiqu()（异步接口，主动从文本中提取标签）

```
读取 AI 配置文件 duqu_aipeizhi()
       │ None → 返回 None
       ▼
构建系统提示词 goujian_xitongtishici()
(从配置的 xinxi_yingshe.keys() 读取允许的类别)
       │
       ├──────────────────────┐
       ▼                      ▼
  第一次 AI 提取          第二次 AI 提取
  danci_tiqu()           danci_tiqu()
  (相同输入)             (相同输入再来一次)
       │                      │
       └──────────┬───────────┘
                  ▼
       去重合并两次结果
       quchong_hebing()
       按(名称+类别)去重
                  │ 空 → 返回 None
                  ▼
          验证标签 yanzheng_biaoqian()
                  │
                  ▼
             返回结果
```

## 入口二：zhixing()（同步接口，ReAct 循环中 AI 调用工具时执行）

```

       ▼

```

## 共用验证流程：yanzheng_biaoqian()

```
传入标签列表 + AI配置
       │
       ▼
┌─────────────────────────────────────┐
│ ① 占位符检查 jiancha_zhanweifu()     │
│   检测："无","暂无","N/A","未提及",   │

└──────────────────┬──────────────────┘
                   ▼

│   只保留配置中允许的类别               │

                   ▼
┌─────────────────────────────────────┐

  Tiqujieguo { chenggong: true, ... }
```

## 来源验证子流程：pianduan_pipei()（片段匹配算法）

```
标签内容
   │
   ▼
整体匹配原文？ ── 是 → ✅ 通过
   │ 否
   ▼
按分隔符拆分为片段（；、，;,\n）
   │
   ▼
对每个片段：
  1. 去除前缀连接词（"与","和","及","向"）
  2. 整个片段在原文中？→ ✅
  3. 片段 < 4字符？→ 直接检查是否在原文中
  4. 片段 >= 4字符？→ 滑动窗口(窗口=4字符)
     任一子串在原文中 → ✅
   │
   ▼
所有片段都通过 → ✅ 匹配成功
任一片段失败   → ❌ 匹配失败
```

## 核心设计思路

- 异步入口 `tiqu()` 做了两次独立 AI 提取再去重合并，提高召回率和稳定性
- 同步入口 `zhixing()` 多了一步来源验证（`yanzheng_biaoqian_laiyuan`），因为这个入口是 AI 在 ReAct 循环中自己调用的，需要防止 AI 伪造标签内容
- 片段匹配用了滑动窗口策略，允许 AI 对原文做合理拼接重组，但要求至少有 4 字符的子串能在原文中找到，平衡了灵活性和防伪造
